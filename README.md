2.1 MAC(Multiply-Accumulate) 연산 원리

Python
for i in range(m1.size):
    for j in range(m1.size):
        total += m1.get_value(i, j) * m2.get_value(i, j)
MAC 연산은 두 행렬의 동일 위치 원소를 곱해 누적하는 과정입니다. 이는 두 패턴 간의 공간적 일치성을 측정하는 것으로, 곱해지는 두 값이 모두 클수록 누적 합이 커집니다. 따라서 점수가 높을수록 필터의 특징과 패턴이 기하학적으로 더 유사하다고 판단합니다.

2.2 부동소수점 오차(Epsilon) 발생 원인

컴퓨터는 실수를 2진법(IEEE 754 표준)으로 저장합니다. 0.1과 같은 십진수는 이진수로 무한소수가 되어 미세한 **반올림 오차(Truncation Error)**가 발생합니다. 수만 번의 누적 연산 시 이 오차가 커져 a == b 연산이 실패할 수 있으므로, epsilon(1e-9) 이내의 차이는 동일한 값으로 간주하는 안전장치가 필수적입니다.

3.1 측정 환경 및 방법

측정 구간: measure_time 함수 내에서 mac_operation이 수행되는 순수 연산 구간만 측정합니다. 파일 I/O 및 출력 시간을 배제하여 알고리즘 순수 성능을 도출했습니다 (기준 13)

분석: 데이터 크기가 5에서 25로 5배 증가할 때, 연산 횟수는 25에서 625로 25배(5^2) 증가합니다. 이는 이중 반복문 구조로 인해 **시간 복잡도가 O(N^2)**임을 수치적으로 증명합니다.

4.2 1000x1000 대규모 연산 시의 문제

메모리 병목: 100만 개의 요소를 가진 리스트는 파이썬 객체 구조상 메모리에 산재되어 **캐시 미스(Cache Miss)**를 유발합니다.

해결책: NumPy를 사용하여 데이터를 연속된 메모리 레이아웃(C-contiguous)으로 배치하면 CPU 캐시 효율이 극대화되며, SIMD 명령어를 통한 병렬 처리가 가능해집니다.

4.3 CPU vs NPU 병목 현상

CPU (직렬 처리): 명령어를 하나씩 해석하고 순차적으로 실행하는 **제어 오버헤드(Control Overhead)**가 주원인입니다. 연산 자체보다 "어디를 읽을지, 다음 명령은 무엇인지" 결정하는 데 시간이 더 걸립니다.

NPU (병렬 처리): 수천 개의 ALU가 동시에 연산하므로 제어 오버헤드는 적으나, 이 수많은 연산기에 데이터를 동시에 밀어 넣어주지 못하는 **메모리 대역폭(Memory Bandwidth)**이 성능의 한계치(Roofline)가 됩니다.
